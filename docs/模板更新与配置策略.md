# FOG GitHub 更新与统一配置策略

> 状态：执行规则草案
> 适用范围：FOG GitHub 模板仓库、同事本地工作副本、`tools/fog.py`

管理员发布流程见 `docs/管理员维护说明.md`。

## 一、核心原则

后续更新以 GitHub 为主：共享 Skill、工具脚本和文档通过 `git pull --ff-only` 更新；同事自己的 `config/fog_config.yaml`、`workspace/` 业务数据、缓存和运行报告不提交、不覆盖。

不要让同事直接改共享 Skill 代码来表达个人差异。个人差异必须优先进入 `config/fog_config.yaml`，真正需要改共享代码时，走 Git 分支和冲突解决。

## 二、同事首次安装

推荐方式：

```powershell
git clone <FOG GitHub 仓库地址> FOG
cd FOG
.\tools\windows\install.ps1
```

首次运行会创建 `config/fog_config.yaml`。同事填写个人账号、token、目录、图片 API Key 后，再次运行：

```powershell
.\tools\windows\install.ps1
.\tools\windows\check.ps1
```

压缩包仍可作为过渡交付，但不推荐作为长期路径。普通 zip 没有 `.git` 历史，后续不能自然 `git pull`；如果首发用了 zip，后续最好重新 `git clone` 一份，再迁移本地 `config/fog_config.yaml` 和 `workspace/`。

## 三、同事更新

标准更新命令：

```powershell
git status --short
git pull --ff-only
python tools/fog.py migrate-config
python tools/fog.py init
python tools/fog.py check
```

自然语言触发可以写成：

```text
帮我更新 FOG 到最新版本。先检查本地有没有改动；没有改动再 git pull --ff-only，然后迁移配置、初始化目录、运行检查。
```

如果 `git status --short` 有输出，先不要 pull。需要判断改动属于哪类：

| 类型 | 处理方式 |
|------|----------|
| `config/fog_config.yaml`、`workspace/`、缓存、报告 | 这些本来就不应提交；确认 `.gitignore` 生效后忽略 |
| 同事为了个人目录、账号、Key、阈值、默认对接人而改 Skill | 回退代码改动，把差异迁入 `config/fog_config.yaml` |
| 同事确实改了共享 Skill 代码 | 先提交到本地分支，再 `git fetch` / `git rebase origin/main`，只在 Git 报冲突时读取冲突文件融合 |

不要在每次更新前全量读代码做人工融合。正确边界是：Git 先判断是否能 fast-forward；只有出现冲突，才读冲突文件、相关配置和对应 diff。

## 四、统一配置

同事只直接维护一份真实配置：

```text
config/fog_config.yaml
```

模板仓库只提交示例：

```text
config/fog_config.yaml.example
```

所有后续共享 Skill 如需读取配置，必须直接读取 `config/fog_config.yaml`。旧的 `.workbuddy/skills/*/assets/config.yaml` 不再作为运行入口，也不再由初始化流程生成。

当前根配置已覆盖：

- `database`：共享日报等能力可选 PostgreSQL 配置
- `lx_shujuku`：dataReporting 账号与查询限制
- `lx_txdocs`：个人版腾讯文档根目录与 OpenAPI token
- `lx_txsaasdocs`：腾讯文档企业版/SaaS API 域名、自建应用 token 端点和凭证
- `lx_zhutichaibiao`：拆表工作目录、默认对接人、字段规则
- `lx_haibao`：图片 provider、模型、API Key、输出目录
- `lx_dapanribao`：日报输出、企业版腾讯文档目标和阈值
- `lx_nongfu`：农服协作工作目录、默认对接人、品牌城市定位键、日常信息表命名和回填安全规则

不分享给同事的内部 Skill 不写入 `config/fog_config.yaml.example`。当前内部配置放在本地 `config/personal_config.yaml`，例如 `lx_lxdataimport`、`lx_hhgongbu`，该文件不提交、不导出。

`tools/fog.py` 是新的普通工具入口：

```powershell
python tools/fog.py init
python tools/fog.py check
python tools/fog.py migrate-config
```

`lx-init` 仅保留为兼容旧入口；它不再负责生成各 Skill 的 `assets/config.yaml`。

## 五、新增 Skill 准入规则

准备分享给同事的 Skill 必须先满足：

1. 个人化信息全部配置化：账号、token、本机路径、默认对接人、输出目录、阈值、外部服务开关不能写死在脚本里。
2. 配置入口统一：共享 Skill 的新增配置项写入 `config/fog_config.yaml.example`，运行时直接读取 `config/fog_config.yaml`。
3. 分发边界清楚：代码和示例文件可提交；真实配置、缓存、证据包、业务输出必须在 `.gitignore` / `protected_paths` 中。
4. 写操作可预览：发布、上传、外部 API 调用、会产生费用的动作必须支持 `--dry-run` 或显式确认。
5. 文档同步：`AGENT.md`、本文件、`config/template_manifest.yaml`、对应 `SKILL.md` 必须同时说明配置项、依赖和安全边界。

不分享的写库 Skill 不加入模板导出白名单；它们的个人配置写入本地 `config/personal_config.yaml`。

## 六、发布与导出

长期推荐直接维护 GitHub 仓库。发布版本时打 tag / Release，同事通过 `git pull --ff-only` 获取。

如果需要继续产出 zip，必须由源工作区白名单导出，不手工复制：

```bash
python tools/export_template.py --output "/path/to/p-fog-template" --clean
```

预览导出：

```bash
python tools/export_template.py --output "/path/to/p-fog-template" --dry-run
```

禁止发布：

- `config/fog_config.yaml`
- `config/database.yaml`（历史本地文件，不再作为 Skill 默认配置入口）
- `config/码表/*.xlsx`
- `.workbuddy/skills/*/assets/config.yaml`（历史本地文件，不再作为 Skill 默认配置入口）
- `.workbuddy/skills/lx-haibao/output/`
- `.workbuddy/skills/*/assets/*cache*.json`
- 任何 API Key、access_token、refresh_token、密码
- `workspace/` 下除 `.gitkeep` 外的任何业务文件
- `.venv/`、`__pycache__/`、`*.pyc`、`*.zip`、`.DS_Store`
